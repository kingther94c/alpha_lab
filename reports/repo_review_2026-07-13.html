一页结论

这是一个研究纪律明显高于普通个人量化仓库的项目，但“研究做得认真”和“执行层已经可以托付真实资金”是两件事。当前最大风险不在策略想法，而在订单闭环和研究结果的经济口径。

结论：研究可继续，demo / paper 可有条件运行，真实资金执行应暂停。

研究与执行两条腿的边界清楚，207 项测试通过，核心向量回测默认滞后一根 bar，holdout 与决策记录也有真实约束；但执行器存在“目标归零却不会平仓”、下单回执被当成成交、实际持仓未参与风险上限计算等闭环缺口。与此同时，P7 v3.1 已没有未触碰的 OOS 窗口，融资成本仍未成为通用回测引擎的一等公民。

目标归零不会生成平仓单

计划层和 broker 层都过滤零权重，已有仓位可能在策略转空仓后继续暴露。

“已提交”被记录为“已完成”

没有订单 ID、成交量和仓位回查；半边成交仍会锁到次日。

必须先修的三件事

这三项应作为 stop-the-line 工作，不需要等待策略迭代，也不应被新功能挤占。

CRITICAL · EX-01

显式零目标被丢弃，策略无法可靠平仓

实盘阻断

业务影响：策略从持仓切换到空仓时，执行计划为空，broker 直接返回；历史仓位可能继续承受市场波动，而系统界面看起来像“没有需要交易”。

事实 runner.py:52–60 只保留非零 legs； binance.py:81–99 再次跳过零权重，并对空计划提前返回。离线复现：rows=0, empty=True, rebalance_on_empty_returns=[]。

可行修复：计划必须从“策略完整交易域 ∪ 当前实际仓位”构建；零目标要保留，订单量按 target - current 计算。任何当前非零、目标为零的 leg 都必须生成 close delta。

验收：模拟账户先持有 BTC/ETH，再输入全零目标；必须生成平仓单，成交后实际仓位在 dust tolerance 内为零。覆盖 spot、perp、混合、最小下单量和部分成交。
CRITICAL · EX-02

下单提交被当作成交完成，部分成交可锁到次日

状态不可信

业务影响：一条腿失败、另一条腿提交后，runner 仍可能写入当天已 rebalance，并返回 ok。系统没有订单 ID、filled quantity、平均成交价和最终仓位的权威记录。

事实 binance.py:101–112 仅调用 create_order 后追加 tuple； runner.py:105–108 只要有 placed 项就写 rebalance 日期；runner.py:151–155 注释已承认 partial-fill latch。

可行修复：先落订单 intent 和幂等 clientOrderId，再提交；逐单保存交易所 order ID、状态、成交量、均价、错误；全部提交后重新拉取实际持仓，按容差决定 COMPLETE / PARTIAL / FAILED。只允许 COMPLETE 推进日锁。

验收：用 fake broker 注入第二腿超时、第一腿半成交、重复回调、进程中断；重启后不得重复超额下单，并能继续完成或安全回滚到目标容差。
CRITICAL · EX-03

实盘双重确认只在 CLI 完整，直接调用 broker 可绕过一层

安全边界

业务影响：CLI 同时要求环境变量和显式参数，但 Python 调用者可直接构造 live broker；broker 自己只验证 CONFIRM_LIVE=YES。这不满足仓库声明的“双钥匙”原则。

事实 cli.py:31–36 有双门； binance.py:44–46 只检查环境变量。UI 当前固定从 demo 启动，这是优点，但不能替代底层门禁。

可行修复：live broker 构造时必须接收无默认值的 LiveAuthorization 能力对象；该对象仅能由通过 CLI 显式 flag + 环境变量校验的入口创建。所有测试和 UI 默认继续使用 demo。

验收：直接 Python 调用、缺 flag、缺环境变量、demo key 误用于 live、live key 误用于 demo 均 fail closed；只有两项确认同时存在时才允许连接，且测试不触网。

值得保留并继续强化的部分

本次审查不是“推倒重来”。下面这些基础已经有实际价值，后续改造应围绕它们增量演进。

研究 / 执行边界清楚

alpha_lab 不反向依赖 quant_bot_manager；执行层可以复用研究层目标权重，方向正确。

通用回测默认滞后一根 bar

vector.py:117–160 对目标权重做 lag，降低同收盘信号吃同收盘收益的常见错误。

holdout 有机械约束

数据层不仅有文档约定，还有 release 状态、审计记录与相应测试，优于仅靠研究者自觉。

执行状态有 WAL 和单飞锁

SQLite WAL、跨进程 rebalance lock、drawdown 自动 halt 已经搭出正确骨架，适合继续补强而不是换架构。

决策记录能承认失败

P6 主动修正 cost-of-cash，P7 记录 2026 弱化，QQQ 研究限制 paper only；这类诚实比更多策略更有价值。

基本质量底座可用

完整测试套件 207 项通过；高置信 secret 扫描无命中，本地文档链接全部有效，raw/private 目录也有忽略规则。

可信度不是一个总分。需要分别看时序、样本外、成本、样本宇宙和结果复现。当前最强的是信号时序骨架，最弱的是“修正后策略还有没有未触碰验证集”。

R-01 · OOS

P7 v3.1 没有新的 untouched OOS

2026 holdout 暴露问题后，v3.1 的修复在截至 2025 的样本上做选择；这意味着原 2025 OOS 已变成开发样本，而 2026 结果也已经被看过。流程很透明，但证据身份必须重标。

P7 决策记录 与 fix_s5.py:43–44, 157–175。

建议：2025 标为 development，2026 标为 released-holdout / post-event validation；冻结下一段前向期，只允许监控，不再回头调参。

R-02 · Economics

融资成本不是回测引擎的一等公民

合同说 net return 应包含 cost of cash，但通用 vector engine 只统一扣交易、滑点和 funding；crypto / congress 再各自手工 overlay。这让遗漏变得容易，P6 已证明影响可能很大。

research_artifacts.md:205–210； vector.py:79–177； crypto_book.py:188–190。

建议：让 BacktestResult 明确输出 trading / funding / financing / idle-cash 分项，并按部署资本或保证金口径收费。

R-03 · API footgun

全样本 z-score / winsorize 默认可被误用

transforms.py:18–33 默认可沿时间轴使用全样本均值、标准差和分位数。当前未发现 accepted P7 直接调用，因此不是现有结果的已确认泄漏，但作为可复用 API 的默认行为不安全。

建议：把横截面、rolling、expanding 和 diagnostic-only 版本拆成清晰命名；新增“截断数据后历史结果不变”的 truncation-invariance 测试。

R-04 · Survivorship

Universe CSV 与自身合同不一致

country_etf_universe.csv 和 us_sector_etf.csv 缺少合同要求的生效日期 / 版本字段；当前成分被投射到历史研究会产生 survivorship 风险。

Universe 合同； country_etf_universe.csv； us_sector_etf.csv。

建议：新增 schema + point-in-time validator，至少强制 start_date、版本、重复项与覆盖期检查。

泄漏审计结论：自动扫描器在 crypto 与 src/scripts 中报告的最高级命中，大部分是“前向收益作为研究标签”和“诊断性 bootstrap 分位数”，经人工追踪不是交易信号泄漏。当前 accepted P7 未确认存在 same-bar 或 future-feature 泄漏；但 P7 v3.1 的 OOS 已被污染、融资成本口径不完整，所以仍不能把 headline performance 当作 execution-grade 证据。旧 DBMF sizing notebook 已在项目 backlog 自报 full-sample volatility normalization，修复重跑前不应作为可交易绩效。

执行安全：骨架存在，但状态闭环尚未完成

当前执行腿更像“可靠的 demo 控制框架原型”，还不是交易账本。关键缺口不是界面，而是 intent → order → fill → position → risk → audit 的闭环。

EX-04 · Risk basis

gross cap 检查目标计划，不检查实际 / 预计持仓

runner.py:60, 102–104 的风险检查基于 plan。若旧仓位因零目标被丢弃，目标 gross 看似合规，真实 gross 仍可能超限。

修复：按当前仓位 + 拟执行 delta 计算 projected position，并在成交后再次断言实际 gross / net / per-leg 风险。

EX-05 · Credentials

允许不完整 venue 凭据，后续却同时使用 spot 和 futures

binance.py:48–65 接受任一组凭据；binance.py:129–136 的 mark-to-market 又无条件访问两端。当前 P7 需要二者。

修复：每个 bot 声明 required venues / legs，连接前完成整组 key+secret 和权限预检，任何缺口在下单前 fail closed。

EX-06 · Locking

过期锁被新 owner 接管后，旧 owner 仍能删除它

store.py:148–171 支持 600 秒后 steal，但 release 不比较 owner。旧进程晚到的 release 可清除新进程的锁。

修复：token compare-and-delete；长流程 heartbeat；新增“stale steal 后旧 owner release”并发测试。

EX-07 · Process

stop 根据旧 PID 强杀，可能杀错进程或截断审计

state.py:91–103 未校验进程启动 token / command line 就 taskkill。PID 重用和下单后审计前被杀都可能造成异常。

修复：持久化 PID + start token + command identity，先发优雅停止信号并等待，再按身份校验后强杀。

工程健康：测试过关，但“完成标准”没有真正成为门禁

核心代码并非失控；主要问题是规则写得对，却没有机械执行。短期不需要引入重型平台，只需把现有约定变成 fail-fast 检查。

E-01 · Config

配置解析 fail-open

schema.py:29–58 会静默丢弃未知键，并用 Python bool() 转换字符串。离线复现：{"max_gros": 0.1, "paused": "false"} 得到默认 max_gross=2.0 且 paused=True。

修复：拒绝未知键、严格解析 bool、拒绝 NaN/inf，错误消息包含配置来源与字段名。

E-02 · Packaging

可编辑安装不等于可分发安装

pyproject.toml:53–54 的 wheel packages 仅包含 src/alpha_lab，遗漏 quant_bot_manager；依赖也未声明执行腿使用的 ccxt、streamlit。.env.example 还缺 demo key 名并指向不存在脚本。

修复：wheel 纳入两包；增加 execution optional extra；构建 wheel 后在干净环境做 import/CLI smoke test；同步示例配置。

E-03 · Quality gate

207 项测试通过，但 Ruff 门禁失败

ruff check . 报 162 项；缩到 src tests 仍有 26 项，其中 14 项可自动修复。规则要求任务完成前通过 Ruff，目前实际流程与规则脱节。

修复：先清核心 26 项；包和测试作为 blocking lint，notebook/report scripts 使用独立、明确的 lint 范围，不用一次大扫除掩盖功能改动。

E-04 · Notebook hygiene

两本 notebook 带输出进入仓库

01_fed_rate_decisions.ipynb 有 5 个输出 / 6 个 execution count，02_iran_war.ipynb 有 3 个输出 / 4 个 execution count，违反提交前清输出规则。

修复：用 pre-commit 或现有脚本做只读检查；发现输出直接失败，清理动作仍由提交者明确执行。

E-05 · Complexity

当前 QQQ 工作区有几个过长函数

未提交的 run_cash_secured_put_backtest 约 351 行；qqq_putwrite_study.py 约 1,462 行，其中报告、图表、主流程函数各自 200–300 行。测试已通过，但会增加审阅成本。

修复：提交前只拆三类稳定边界：simulation state transition、policy specification、report serialization；不引入抽象基类或插件系统。

E-06 · Agent governance

技能目录存在双份且已有漂移

.claude/skills 被跟踪，.agents/skills 未跟踪且内容已有约 155 行新增 / 144 行删除的差异。团队可能在不同入口执行不同规则。

修复：选一个 canonical source，另一目录自动生成或做同步检查；不要人工双写。

建议的 90 天路线图

顺序遵循“先让状态真实，再让研究口径可比，最后提升产能”。不建议此时增加策略数量或 broker 数量。

0–7 天 · Stop the line

封住可能留下真实仓位的路径

  • 保持 live mode 禁用；不弱化现有任何门禁。
  • 修复 explicit-zero / current-position delta 计划，并补 spot/perp/dust 测试。
  • 建立 per-leg order result 与成交后 position reconciliation；只有 COMPLETE 推进日锁。
  • 风险上限改为 projected + actual 双检查。
  • live capability 下沉 broker；required venue 凭据预检。
第 2–4 周 · 让规则可机械执行

统一经济口径和 fail-fast 契约

  • BacktestResult 增加 trading / funding / financing / idle cash 分项。
  • 重标 P7 样本身份并冻结新的 forward-only 观察期。
  • 严格配置解析；完成 wheel / execution extra / demo env 修复。
  • 替换不安全 transforms 默认，增加 truncation-invariance 测试。
  • 清理 src/tests 26 项 Ruff，并加入 notebook-output 检查。
第 2–3 月 · 提升复现与运营能力

把可信度证据自动沉淀

  • 研究运行 manifest + 决策登记表，记录数据、代码、样本和成本身份。
  • Universe / data quality validator，优先覆盖 country/sector ETF。
  • qbm doctor 无下单运行前自检。
  • shadow execution drift monitor 与 graceful process lifecycle。
  • 在提交当前 QQQ 工作前拆分长状态机和报告代码。

实际可行的新功能建议

这些不是“加 AI”“做平台”式泛泛建议，而是直接复用现有 SQLite、BacktestResult、decision records 和配置结构的小步能力。工期按熟悉仓库的 1 名工程师估算，不包含外部交易所联调等待。

优先级 P03–5 人日

1. Reconciliation-first Executor

把当前“发订单的 runner”升级为“以实际持仓收敛为目标的状态机”，一次解决零目标、部分成交、幂等和重启恢复。

  • 持久化 intent、clientOrderId、exchange order ID、fills、position snapshots。
  • 状态：PLANNED → SUBMITTED → PARTIAL → COMPLETE / FAILED。
  • 仅在实际仓位进入容差后标记完成。

验收：5 类故障注入全部安全恢复；重复运行不增加超额敞口。

优先级 P11–2 人日

2. qbm doctor 运行前自检

在任何 runner 启动前，无下单地检查配置、依赖、凭据来源、venue 完整性、市场精度、时钟与数据新鲜度。

  • 默认只检查 demo；live 仍需双重确认且 doctor 不下单。
  • 输出人类可读报告与机器可读 exit code。
  • 复用 BotConfig 和 broker market metadata，不建新配置框架。

验收：当前 .env.example 缺口、单边凭据和未知配置键均被明确报错。

优先级 P13–5 人日

3. 回测经济成本账本

把交易成本、funding、资金占用、借贷成本和 idle cash 收益统一纳入引擎，消除策略各自补公式造成的漂移。

  • BacktestResult 保留 gross 与每项 cost series。
  • 支持 cash-secured、futures margin、long/short collateral 三种最小模型。
  • 自动输出 0.5× / 1× / 2× 成本敏感度。

验收：P6 修正可由通用引擎重现；不同策略报告可核对到同一净收益恒等式。

优先级 P13–5 人日

4. Research Run Manifest + Decision Registry

每次正式研究自动生成一个小型 JSON manifest，避免“这张图用的哪版数据、哪段 OOS、工作区是否脏”靠记忆。

  • 记录 git SHA / dirty、data hash/as-of、universe version、参数、split 身份。
  • 记录成本口径、leak scan verdict、关键指标和 decision record 链接。
  • 从 manifests 生成管理层索引，不替代现有 markdown 决策记录。

验收：任何 headline 指标可一跳定位到代码、数据、成本和样本身份；released holdout 会自动标为已污染。

优先级 P12–4 人日

5. Universe / Data Quality Validator

把已有合同变成加载时的机械校验，优先解决 country / sector ETF 的 point-in-time 与数据缺口风险。

  • 检查 required fields、start/end date、重复、ticker change、覆盖期和 stale gaps。
  • 输出 universe fingerprint 写入 manifest。
  • 按“error / warning / diagnostic”分级，避免阻断探索性 notebook。

验收：当前两份不完整 universe CSV 会失败并给出可操作字段清单。

优先级 P23–5 人日

6. Shadow Execution Drift Monitor

在 demo/paper 阶段持续比较目标、订单、成交和实际仓位，让执行误差在进入实盘讨论前被量化。

  • 按 leg 记录 target/current/drift、延迟、拒单、滑点与未成交。
  • 超容差只 halt + alert，不自动加风险追单。
  • 先写 SQLite / HTML 日报，暂不引入消息平台或监控栈。

验收：连续 30 个 paper rebalance 有完整对账，任何残留仓位和部分成交都可追溯。

不建议现在做：新 broker 插件体系、微服务化、通用策略 DSL、实时大屏、自动实盘开关、复杂配置注册中心。当前瓶颈是正确性闭环，不是扩展点数量。

范围、方法与限制

审查范围

  • src/alpha_lab：数据、特征、分析、回测、组合、ML/CV。
  • src/quant_bot_manager：broker、runner、risk、store、CLI/UI。
  • tests、notebooks、scripts、configs、contracts、decision records、skills。
  • 基于 main commit 304b9053…，并包含当前未提交的 QQQ/congress/agent 工作。

工作区注意事项

审查开始时工作区已非 clean，包括 congress 配置、QQQ put-write、congress analytics/tests、skills 等未提交内容。本报告只读取和分析这些内容，没有改写或整理它们。

因此，针对未提交代码的结论是“提交前建议”，不能当作 main 上已发布缺陷。

方法

  • 架构与依赖方向检查。
  • 测试、Ruff、secret、本地链接、notebook 输出与规模检查。
  • 项目 leakage skill 静态扫描 + 手工 cardinal-sin checklist。
  • 执行状态机、门禁、锁、配置、打包的路径级追踪。
  • 两个高风险行为的最小离线复现。

限制

  • 没有使用真实交易所 sandbox / demo API 做端到端成交测试。
  • 没有重跑全部数据下载或所有 notebook，因此未验证外部数据可重复性。
  • 静态泄漏扫描是启发式；最终 verdict 结合人工数据流追踪。
  • 工期是单工程师的相对估算，未含外部 API 与市场规则变更。
审查判断边界：“未发现已确认泄漏”不等于“绩效已可投入资金”；样本外身份、成本、滑点、容量和执行闭环必须同时满足。本文不是投资建议，也不授权任何真实资金交易。
